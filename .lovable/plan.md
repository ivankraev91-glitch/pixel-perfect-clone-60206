
# Парсинг Яндекс.Карт: архитектура

## Идея

Вместо платного Geosearch API дёргаем внутренний JSON-эндпоинт Яндекс.Карт `https://yandex.ru/maps/api/search` (его использует сам сайт карт). Он отдаёт ровно ту же выдачу, что видит пользователь — то есть «настоящие» позиции в локальном паке. Запросы идут **с привязкой к координате** (`ll`, `spn`, `z`), что нам и нужно для проверки позиций по гео-точкам.

Защита со стороны Яндекса:
- Капча (SmartCaptcha) — лезет при подозрении на бота.
- IP-рейт-лимиты — один IP быстро попадает в бан.
- Cookie-сессии (`yandexuid`, `Session_id`) — без них быстрее ловится капча.

Поэтому архитектура должна закрывать три вещи: **резидентные прокси РФ**, **решение капчи**, **очередь и кэш**, чтобы под нагрузкой не упереться в лимиты и не ловить бан.

## Стек интеграций (всё через секреты Lovable Cloud)

1. **Прокси-провайдер РФ** (резидентные/мобильные, ротация по запросу).
   На выбор: ProxyLine, Proxy6, Proxy-Sale, Astroproxy. Нужны RU IP, формат `http://user:pass@host:port`.
   Секреты: `RU_PROXY_URL` (один URL ротатора) или `RU_PROXY_LIST` (список через запятую).

2. **Антикапча**: 2Captcha / RuCaptcha / CapMonster.
   Поддерживают Yandex SmartCaptcha (метод `yandexsmartcaptcha`).
   Секрет: `CAPTCHA_API_KEY`, `CAPTCHA_PROVIDER` (`2captcha` | `capmonster`).

3. **Лимиты/очередь**: Postgres-таблица `scrape_jobs` + edge-функция-воркер, дёргаемая по cron (pg_cron) каждую минуту. Это даёт контроль скорости и retry без внешних сервисов.

## Архитектура

```text
                    ┌──────────────┐
   UI «Проверить» ─►│ enqueue-check│──► scrape_jobs (pending)
                    └──────────────┘
                           │
                pg_cron каждую минуту
                           ▼
                    ┌──────────────┐    proxy + cookies
                    │ scrape-worker│──────────────────────► yandex.ru/maps/api/search
                    └──────────────┘    │
                           │            ▼ (если HTML с капчей)
                           │     ┌───────────────┐
                           │     │ solve-captcha │──► 2captcha
                           │     └───────────────┘
                           ▼
                       checks (результат)
                           │
                           ▼
                       UI обновляется (realtime)
```

## Поток одной проверки

1. Пользователь жмёт «Проверить» → создаётся запись в `scrape_jobs` со статусом `pending` (org_id, keyword_id, geopoint_id).
2. Воркер берёт до N задач (батч), для каждой:
   - Берёт следующий прокси из пула (round-robin, помечает «cooling» на 30 сек).
   - Тянет/переиспользует cookie-сессию для этого прокси (таблица `scrape_sessions`).
   - GET `https://yandex.ru/maps/api/search?text=<keyword>&ll=<lon,lat>&spn=0.02,0.02&z=15&lang=ru&csrfToken=…`.
   - Если ответ JSON с результатами → парсим, ищем `oid` организации, считаем позицию.
   - Если ответ HTML/редирект на капчу → отправляем sitekey в 2Captcha, повторяем запрос с токеном, обновляем cookies.
   - Если 429/403 → помечаем прокси как «бан 1 час», ставим задачу обратно в очередь с `attempts+1`.
3. Результат пишется в `checks`, задача → `done`. Если `attempts >= 5` → `failed` с текстом ошибки.
4. UI слушает `checks` через Supabase Realtime — позиция появляется без перезагрузки.

## Что строим (этапы)

### Этап 1. База очереди и сессий
Миграция:
- `scrape_jobs` (id, user_id, org_id, keyword_id, geopoint_id, status, attempts, error, created_at, started_at, finished_at) + RLS «свои задачи».
- `scrape_sessions` (id, proxy, cookies jsonb, last_used_at, banned_until) — service-role only.
- Включить `pg_cron` и `pg_net`, добавить расписание: каждую минуту вызывать edge-функцию `scrape-worker` через `net.http_post` с service-role ключом.

### Этап 2. Edge-функции
- `enqueue-check` — заменяет текущую `check-position`. Просто кладёт задачу в очередь, возвращает `job_id`. Сохраняет дневной лимит на пользователя (например, 50/день).
- `scrape-worker` — основная логика: берёт батч (5 задач), параллельно обрабатывает, использует прокси/капчу. Защищён secret-токеном в заголовке (cron его передаёт).
- `solve-captcha` — внутренний хелпер (можно прямо в воркере функцией): отправляет в 2Captcha, ждёт токен.
- `search-org` — переписать на тот же парсинг (один разовый запрос, без очереди — он не критичен по масштабу).

### Этап 3. UI
- Кнопка «Проверить сейчас» → `enqueue-check` → тост «Поставлено в очередь, обычно 10–30 сек».
- Realtime-подписка на свои `scrape_jobs` и `checks` → автообновление дашборда.
- Индикатор «в очереди / выполняется / ошибка».
- В Settings — счётчик «Проверок сегодня: X / 50».

### Этап 4. Стабильность под масштаб
- **Кэш выдачи**: одинаковый `(keyword, geopoint, ±5 мин)` → отдаём кешированный результат, не дёргая Яндекс. Экономит 60–80% запросов когда у вас будет много пользователей с пересекающимися гео.
- **Адаптивная скорость**: если доля капчи >30% за последние 100 запросов → воркер уменьшает батч и ставит паузу 10 сек между запросами.
- **Здоровье прокси**: таблица `proxy_health` (успехов/ошибок за 1 час). Прокси с success_rate <50% → исключаются автоматически на 6 часов.
- **Алерты**: если очередь >500 задач или 0 успехов за 10 минут → запись в `system_alerts`, видна в админ-странице.
- **Бэкофф**: между ретраями одной задачи 30s → 2m → 10m → 30m → 1h.
- **Отдельный пул прокси для `search-org`** (поиск при онбординге) — он чувствителен к latency, не должен страдать от очереди проверок.

## Юридический момент (один абзац, чтобы было понятно)

Парсинг публичной выдачи Яндекс.Карт **нарушает их пользовательское соглашение**, даже если технически работает. Риски: бан IP-диапазонов, в теории — претензии при коммерческом использовании в больших объёмах. На небольшом масштабе (десятки—сотни проверок в день на пользователя) это типовая практика SEO-сервисов, но если планируется тысячи запросов в час или продажа клиентам — стоит параллельно подключить и платный тариф Яндекса как «легальный fallback».

## Что нужно от вас перед стартом

1. Купить и дать ключи: **прокси РФ** (URL ротатора или список), **2Captcha/RuCaptcha** (API key).
2. Подтвердить дневной лимит проверок на пользователя (предлагаю 50/день).
3. Подтвердить, что готовы к юридическому риску из абзаца выше.

После одобрения плана: я делаю миграцию (этап 1), потом по очереди этапы 2–4, и в конце прошу через `add_secret` ввести ключи прокси и капчи.
